# Moniterra / Wialon: що вже відомо

## 1. Мета інтеграції

Система має один раз на добу:

1. Авторизуватися в Moniterra/Wialon через API-токен.
2. Виконати паливний звіт для кожного автомобіля.
3. Отримати добову статистику та хронологію поїздок.
4. Нормалізувати й записати дані в Supabase.
5. Класифікувати маршрут.
6. Порівняти витрату пального з історичним baseline.
7. Надіслати підсумок та аномалії в Telegram.

Інтеграція використовує Wialon Remote API Reports. Вона не читає сирий
телематичний потік і не замінює Moniterra.

## 2. Підтверджена API-конфігурація

| Параметр | Значення |
|---|---|
| API endpoint | `https://moniterra.services/wialon/ajax.html` |
| Report resource ID | `2217` |
| Fuel report template ID | `5` |
| Fuel report template | `.Отчет по топливу` |
| Trips report template ID | `11` |
| Trips report template | `.Поездки` |
| Geofences report template ID | `1` |
| Geofences report template | `Геозоны` |
| Перевірений unit ID | `3764` |
| Перевірений автомобіль | `01. KA2790BA / AA2544XC` |
| Report object secondary ID | `0` |
| Режим виконання | асинхронний, `remoteExec: 1` |

`reportResourceId`, `reportTemplateId` і `reportObjectId` були отримані з
реального запиту `report/exec_report` у Moniterra.

## 3. Авторизація

Для серверної інтеграції використовується вже створений API-токен:

```text
svc=token/login
params={"token":"<WIALON_TOKEN>"}
```

У відповіді поле `eid` є ID сесії, який далі передається як `sid`.

Властивості сесії:

- `sid` є тимчасовим і не зберігається в конфігурації;
- після п'яти хвилин без запитів сесія може бути завершена сервером;
- після роботи необхідно викликати `core/logout`;
- токен і `sid` не можна логувати або записувати в Markdown.

Раніше опублікований токен потрібно вважати скомпрометованим. Для
production потрібен новий токен із мінімально необхідними правами.

## 4. Підтверджений lifecycle звіту

```text
token/login
  -> отримати eid як sid
report/exec_report { remoteExec: 1 }
report/get_report_status {}
  -> status 1: очікує в черзі
  -> status 2: виконується
  -> status 4: готовий
  -> status 8: скасований
  -> status 16: звіт не знайдено / помилка
report/apply_report_result {}
report/select_result_rows {...}
report/cleanup_result {}
core/logout {}
```

Приклад підтвердженого запуску:

```json
{
  "reportResourceId": 2217,
  "reportTemplateId": 5,
  "reportTemplate": null,
  "reportObjectId": 3764,
  "reportObjectSecId": 0,
  "interval": {
    "flags": 16777218,
    "from": 0,
    "to": 1
  },
  "remoteExec": 1
}
```

Значення `16777218` (`0x1000002`) було сформоване інтерфейсом для
відносного періоду "попередній день". У production краще передавати
явні UNIX timestamps `from` і `to` з `flags: 0`, щоб уникнути залежності
від UI-правил та часового поясу.

## 5. Обмеження сесії Wialon

В одній `sid`-сесії одночасно може існувати результат лише одного
виконаного звіту.

Заборонена модель:

```text
одна sid -> Promise.all(20 x report/exec_report)
```

Допустимі моделі:

1. Один груповий звіт для всього автопарку, якщо доступний відповідний
   шаблон і group ID.
2. Окрема `sid` для кожного паралельного worker.
3. Послідовне виконання звітів в одній `sid`.

Практичний fallback для Vercel: пакети по 4-6 автомобілів, де кожен worker
має власну сесію. Точний рівень паралельності потрібно перевірити
навантажувальним тестом і за реакцією Moniterra API.

## 6. Структура паливного звіту

### 6.1 Добова статистика

`reportResult.stats` повертає готові агреговані показники:

| Показник | Приклад |
|---|---:|
| Пробіг у поїздках | `250 km` |
| Міський пробіг | `9.86 km` |
| Заміський пробіг | `240 km` |
| Максимальна швидкість | `91 km/h` |
| Середня швидкість | `75 km/h` |
| Початковий рівень пального | `471 l` |
| Кінцевий рівень пального | `419 l` |
| Кількість заправок | `0` |
| Обсяг заправок | `0 l` |
| Кількість зливів | `0` |
| Обсяг зливів | `0 l` |
| Витрачено за ДУТ | `52 l` |
| Середня витрата за ДУТ | `20.92 l/100 km` |
| Кількість стоянок | `5` |

Для тестового дня показники узгоджуються:

```text
471 l - 419 l = 52 l
```

Highway ratio:

```text
240 / 250 = 0.96, тобто приблизно 96%
```

Парсер має використовувати числові raw values там, де вони доступні.
Якщо статистика повертається лише як форматований текст, одиниці потрібно
перевіряти та нормалізувати окремо.

### 6.2 Таблиця хронології

Підтверджена таблиця:

| Поле | Значення |
|---|---|
| `tableIndex` | `0` |
| `name` | `unit_chronology` |
| `label` | `Хронология по топливу` |
| `rows` | залежить від дня |
| `level` | `1` |
| `columns` | `8` |

Колонки:

| Індекс | Header | Призначення |
|---:|---|---|
| `c[0]` | Тип | `Trip`, заправка, злив або інша подія |
| `c[1]` | Початок | час, координати, unit ID |
| `c[2]` | Кінець | час, координати, unit ID |
| `c[3]` | Тривалість | форматований час |
| `c[4]` | Початкова позиція | адреса й координати |
| `c[5]` | Кінцева позиція | адреса й координати |
| `c[6]` | Опис | наприклад `Mileage: 223 km` |
| `c[7]` | Нотатки | довільний текст |

Запит рядків:

```json
{
  "tableIndex": 0,
  "config": {
    "type": "range",
    "data": {
      "from": 0,
      "to": 499,
      "level": 1,
      "flat": 1,
      "rawValues": 1,
      "unitInfo": 1
    }
  }
}
```

Для великих таблиць рядки потрібно читати сторінками.

## 7. Підтверджений приклад маршруту

За тестовий день отримано чотири поїздки:

1. Canterbury, UK -> Dover, UK: `26 km`.
2. Маневр у Dover: `0.86 km`.
3. Маневр у Dover: `0.20 km`.
4. Calais, France -> Nivelles, Belgium: `223 km`.

Сумарно:

```text
26 + 0.86 + 0.20 + 223 = 250.06 km
```

Це збігається з округленим добовим пробігом `250 km`.

Попередня класифікація:

```text
route_tag = UK_EU_INTERNATIONAL
route_key = GB:CANTERBURY>BE:NIVELLES
highway_ratio = 0.96
fuel_consumption = 20.92 l/100 km
```

Розрив Dover -> Calais потрібно трактувати як поромний/транспортний
перехід, а не як відсутню автомобільну поїздку.

Короткі переміщення менше визначеного порогу, наприклад `2 km`, потрібно
позначати як локальні маневри й не використовувати як окремий route key.

### Другий контрольний паливний звіт

Для автомобіля `KA6017IK / AA1197XJ`, unit ID `6221`, за
`2026-06-14` підтверджено:

| Показник | Значення |
|---|---:|
| Пробіг | `391 km` |
| Міський пробіг | `12.16 km` |
| Заміський пробіг | `379 km` |
| Частка заміського пробігу | `96.93%` |
| Максимальна швидкість | `93 km/h` |
| Середня швидкість | `80 km/h` |
| Початковий рівень пального | `775 l` |
| Кінцевий рівень пального | `668 l` |
| Витрачено за ДУТ | `107 l` |
| Середня витрата | `27.42 l/100 km` |
| Стоянки | `3` |
| Заправки / зливи | `0 / 0` |

Перевірка узгодженості:

```text
775 - 668 = 107 l
107 / 391 * 100 = 27.37 l/100 km
```

Невелика різниця з `27.42 l/100 km` пояснюється тим, що Wialon рахує
за внутрішніми неокругленими значеннями пробігу та витраченого пального,
а у статистиці показує округлені `391 km` і `107 l`.

`apply_report_result` повідомляє про `2` рядки в таблиці
`unit_chronology`. Відповідь `report/select_result_rows` підтверджена:

1. De Lutte, Netherlands -> Hoogstraten, Belgium:
   `231 km`, `2:51:57`.
2. Hoogstraten, Belgium -> Oostende, Belgium:
   `161 km`, `2:01:47`.

Попередня класифікація:

```text
route_tag = EU_INTERNATIONAL
route_key = NL:DE_LUTTE>BE:OOSTENDE
countries = NL>BE
```

Сума відображених сегментів становить `392 km`, тоді як точний добовий
пробіг у `reportLayer` становить `391.159 km`. Це очікувана похибка через
округлення кожного сегмента до цілих кілометрів.

## 8. Часовий пояс

Звіт відображає інтервал:

```text
2026-06-11 00:00:00 -> 2026-06-11 23:59:59
```

Відповідні UNIX timestamps:

```text
1781125200 -> 2026-06-10 21:00:00 UTC
1781211599 -> 2026-06-11 20:59:59 UTC
```

Отже, звіт сформований у часовому поясі `UTC+03:00`.

Production cron повинен:

- явно визначати business timezone;
- обчислювати межі попереднього календарного дня в цьому timezone;
- передавати відповідні UNIX timestamps;
- зберігати timestamps у Supabase як UTC;
- окремо зберігати `report_date`.

## 9. Що вже достатньо для MVP

Поточний шаблон дозволяє отримати:

- добовий пробіг;
- міський і заміський пробіг;
- витрату пального;
- середню витрату на 100 км;
- кількість та обсяг заправок;
- кількість та обсяг зливів;
- хронологію поїздок;
- адреси та координати початку/кінця;
- unit ID;
- основу для route tagging.

## 10. Що ще потрібно дослідити

1. Список усіх автомобілів та їх `unit ID`.
2. Чи існує unit group і який у неї ID.
3. Чи можна виконати один груповий звіт для всього автопарку.
4. Приклад дня із заправкою та реальний формат такої події.
5. Приклад дня зі зливом та реальний формат такої події.
6. Джерело driver ID/name для рейтингу водіїв.
7. Поведінка API при 4-6 паралельних сесіях.
8. Максимальний типовий час виконання звіту.
9. Остаточний business timezone.
10. Якість адрес і правила нормалізації країн/міст.

## 10.1 Підтверджений звіт `.Поездки`

Для unit ID `6221` за `2026-06-14` підтверджена таблиця:

| Поле | Значення |
|---|---|
| Internal name | `unit_trips` |
| Label | `Поездки` |
| Table index | `0` |
| Rows | `2` |
| Level | `1` |
| Columns | `17` |

Таблиця містить:

- час і адресу початку та завершення;
- тривалість;
- пробіг;
- міський та заміський пробіг;
- середню витрату за ДУТ;
- середню та максимальну швидкість;
- витрачене пальне;
- початковий і кінцевий рівень пального.

Колонки водія у цьому шаблоні немає. Отже, `.Поездки` не вирішує задачу
рейтингу водіїв без додаткового джерела driver ID та історичної
прив'язки водія до автомобіля.

Для завершення аналізу потрібні:

Payload `report/exec_report` підтверджено:

```json
{
  "reportResourceId": 2217,
  "reportTemplateId": 11,
  "reportObjectId": 6221,
  "reportObjectSecId": 0,
  "remoteExec": 1
}
```

Response `report/select_result_rows` для двох рядків `unit_trips`
підтверджено.

Мапінг масиву `c`:

| Індекс | Поле |
|---:|---|
| `c[0]` | час початку та координати |
| `c[1]` | початкова адреса та координати |
| `c[2]` | час завершення та координати |
| `c[3]` | кінцева адреса та координати |
| `c[4]` | тривалість поїздки |
| `c[5]` | загальний час |
| `c[6]` | пробіг |
| `c[7]` | середня витрата за ДУТ |
| `c[8]` | міський пробіг |
| `c[9]` | заміський пробіг |
| `c[10]` | початковий абсолютний пробіг |
| `c[11]` | кінцевий абсолютний пробіг |
| `c[12]` | середня швидкість |
| `c[13]` | максимальна швидкість та координати |
| `c[14]` | витрачено за ДУТ |
| `c[15]` | початковий рівень пального |
| `c[16]` | кінцевий рівень пального |

Підтверджені сегменти:

| Маршрут | Пробіг | Urban / highway | Витрата | Пальне |
|---|---:|---:|---:|---:|
| De Lutte -> Hoogstraten | `231 km` | `4.62 / 226 km` | `28.82 l/100 km` | `66 l` |
| Hoogstraten -> Oostende | `161 km` | `7.54 / 153 km` | `26.17 l/100 km` | `42.01 l` |

Цей звіт корисніший за паливну хронологію для аналітики окремих
сегментів, оскільки одразу повертає витрату, міський/заміський пробіг,
швидкість та рівні пального для кожної поїздки.

## 10.2 Перевірений звіт `Геозоны`

Підтверджений payload для unit ID `6221`:

```json
{
  "reportResourceId": 2217,
  "reportTemplateId": 1,
  "reportObjectId": 6221,
  "reportObjectSecId": 0,
  "remoteExec": 1
}
```

За `2026-06-14` звіт виконався без API-помилки, але повернув:

```text
stats = []
tables = []
attachments = []
```

Це означає одне з двох:

1. Автомобіль не відвідав жодної налаштованої геозони за цей день.
2. Шаблон `Геозоны` не має доступних або правильно налаштованих геозон.

Для MVP не можна залежати від цього звіту. Route tagging будуємо за
адресами та координатами зі звіту `.Поездки`. Геозони залишаємо
додатковим джерелом контексту після перевірки іншого дня або авто.

## 11. Serverless-вимоги

- Один Vercel Cron endpoint.
- Перевірка `Authorization: Bearer <CRON_SECRET>`.
- Lock запуску в Supabase.
- Ідемпотентний upsert за автомобілем і датою.
- `maxDuration` відповідно до тарифу Vercel.
- Timeout кожного HTTP-запиту через `AbortController`.
- Обмежена паралельність, а не необмежений `Promise.all`.
- Retry лише для тимчасових помилок: network error, `429`, `5xx`.
- `cleanup_result` і `logout` у `finally`.
- Частковий збій автомобіля не повинен зупиняти весь автопарк.
- Telegram-звіт має показувати кількість успішних і невдалих автомобілів.
