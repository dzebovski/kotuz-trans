# Як знайти потрібні ID в Moniterra/Wialon

## 1. Безпека перед початком

У DevTools можна побачити токени, cookies та `sid`. Не надсилати їх у
чат, не вставляти у документацію і не публікувати на скріншотах.

Можна передавати:

- API URL без query parameters;
- `reportResourceId`;
- `reportTemplateId`;
- `reportObjectId`;
- unit/group/driver IDs;
- `params` без токена та `sid`;
- анонімізовані API responses.

Не можна передавати:

- `token`;
- `sid`;
- cookies;
- request authorization headers.

## 2. Підготовка Chrome DevTools

1. Відкрити Moniterra.
2. Відкрити `DevTools`:
   - macOS: `Option + Command + I`;
   - Windows/Linux: `Ctrl + Shift + I`.
3. Перейти у вкладку `Network`.
4. Натиснути `Fetch/XHR`.
5. Увімкнути `Preserve log`, якщо потрібно зберегти запити між переходами.
6. Увімкнути `Disable cache`.
7. Очистити список кнопкою `Clear`.

Службові запити на кшталт `events/check_updates`, `avl_evts` і
`avl_hittest_pos` для пошуку report IDs не потрібні.

## 3. Як знайти API endpoint

1. Відкрити будь-який Wialon API request у `Network`.
2. Перейти у `Headers`.
3. Знайти `Request URL`.
4. Взяти частину URL до `?svc=`.

Підтверджене значення:

```text
https://moniterra.services/wialon/ajax.html
```

Не копіювати query string, якщо там є `sid`.

## 4. Як знайти Resource ID, Template ID та Unit ID

### Варіант A: прямий `exec_report`

1. Відкрити розділ `Reports`.
2. Вибрати потрібний template.
3. Вибрати один автомобіль.
4. Вибрати короткий інтервал, наприклад `Yesterday`.
5. Очистити `Network`.
6. У фільтрі ввести:

```text
exec_report
```

7. Натиснути `Execute`.
8. Відкрити:

```text
ajax.html?svc=report/exec_report
```

9. Перейти у `Payload`.
10. Розгорнути `Form Data -> params`.

Приклад:

```json
{
  "reportResourceId": 2217,
  "reportTemplateId": 5,
  "reportObjectId": 3764,
  "reportObjectSecId": 0,
  "remoteExec": 1
}
```

Значення:

- `reportResourceId` - ID ресурсу, де збережено шаблон;
- `reportTemplateId` - ID вибраного шаблону;
- `reportObjectId` - ID вибраного автомобіля або групи;
- `reportObjectSecId` - додатковий об'єкт; для unit-звіту зазвичай `0`.

### Варіант B: `core/batch`

Moniterra може відправити кілька API-команд одним batch-запитом.

1. У фільтрі ввести:

```text
core/batch
```

2. Відкрити `Payload -> params`.
3. Знайти всередині масиву команду:

```json
{
  "svc": "report/exec_report",
  "params": {}
}
```

4. Взяти IDs із вкладеного `params`.

## 5. Як зібрати ID всіх автомобілів

### Ручний спосіб

Для кожного автомобіля:

1. У `Reports` вибрати той самий fuel template.
2. У полі `Object` вибрати автомобіль.
3. Очистити Network.
4. Натиснути `Execute`.
5. Відкрити `report/exec_report`.
6. Записати:
   - `reportObjectId`;
   - повну назву автомобіля;
   - номер тягача;
   - номер причепа, якщо він є в назві.
7. Додати рядок у `02-integration-id-registry.md`.

Не потрібно чекати завантаження всього звіту, якщо мета лише отримати
`reportObjectId`.

### Швидший спосіб через список Units

1. Відкрити розділ `Units`.
2. Очистити Network.
3. Оновити сторінку або виконати пошук.
4. Шукати запити:

```text
core/search_items
```

або:

```text
core/batch
```

5. У `Response` знайти масив `items`.
6. Для кожного елемента виписати:
   - `id`;
   - `nm` або `name`;
   - `cls`, якщо повертається;
   - реєстраційні поля, якщо повертаються.

Перед масовим використанням перевірити один ID через `exec_report`.

## 6. Як знайти ID групи автомобілів

1. У `Reports -> Object` перевірити, чи можна вибрати не автомобіль, а групу.
2. Вибрати групу.
3. Запустити звіт.
4. Відкрити `report/exec_report`.
5. Записати `reportObjectId`.
6. Перевірити, чи присутній `reportObjectIdList`.

Альтернативно:

1. У Network знайти `core/search_items`.
2. Знайти елементи класу unit group.
3. Виписати `id` і назву.

Після цього перевірити, чи template ID `5` підтримує звіт за групою. Не
можна вважати unit ID та group ID взаємозамінними без тестового запуску.

## 7. Як перевірити асинхронний звіт

Після `exec_report` із:

```json
{"remoteExec": 1}
```

Moniterra опитує:

```text
report/get_report_status
```

Статуси:

| Status | Значення |
|---:|---|
| `1` | очікує в черзі |
| `2` | виконується |
| `4` | готовий |
| `8` | скасований |
| `16` | помилка / звіт не знайдено |

Після `status: 4` знайти:

```text
report/apply_report_result
```

Його response містить:

- `reportResult.stats`;
- `reportResult.tables`;
- `reportResult.attachments`;
- `reportLayer`.

## 8. Як знайти tableIndex та структуру таблиць

1. У Network ввести:

```text
apply_report_result
```

2. Відкрити `Response`.
3. Знайти:

```json
{
  "reportResult": {
    "tables": []
  }
}
```

4. Індекс елемента в масиві `tables` є його `tableIndex`.

Приклад:

```json
{
  "name": "unit_chronology",
  "label": "Хронология по топливу",
  "rows": 4,
  "level": 1,
  "columns": 8,
  "header": [
    "Тип",
    "Начало",
    "Конец",
    "Длительность",
    "Нач. положение",
    "Конеч. положение",
    "Описание",
    "Заметки"
  ]
}
```

Оскільки це перший елемент масиву:

```text
tableIndex = 0
```

## 9. Як отримати рядки таблиці

1. В інтерфейсі натиснути потрібну таблицю, наприклад
   `Хронология по топливу`.
2. У Network ввести:

```text
select_result_rows
```

3. Відкрити `Payload -> params`.
4. Перевірити `tableIndex`.
5. Відкрити `Response`.

Рекомендована конфігурація для власного API-клієнта:

```json
{
  "tableIndex": 0,
  "config": {
    "type": "range",
    "data": {
      "from": 0,
      "to": 499,
      "level": 1,
      "flat": 1,
      "rawValues": 1,
      "unitInfo": 1
    }
  }
}
```

Якщо `rows > 500`, виконувати кілька запитів:

```text
0..499
500..999
1000..1499
```

## 10. Як читати `unit_chronology`

Мапінг:

```text
c[0] = тип події
c[1] = початок: час, координати, unit ID
c[2] = кінець: час, координати, unit ID
c[3] = тривалість
c[4] = початкова адреса та координати
c[5] = кінцева адреса та координати
c[6] = опис
c[7] = нотатки
```

Приклад поїздки:

```text
type: Trip
from: Canterbury, UK
to: Dover, UK
mileage: 26 km
unit ID: 3764
```

## 11. Як знайти день із заправкою

1. У fuel report вибирати попередні дні або тижні.
2. Запускати звіт.
3. У `reportResult.stats` перевіряти:
   - `Всего заправок`;
   - `Всего заправлено`.
4. Коли значення більше нуля, відкрити `unit_chronology`.
5. Зберегти:
   - дату;
   - unit ID;
   - весь рядок події;
   - значення `c[0]`;
   - значення `c[6]`;
   - координати й час.
6. Додати дату в реєстр тестових днів.

## 12. Як знайти день зі зливом

Процедура аналогічна заправці. У статистиці перевіряти:

- `Всего сливов`;
- `Всего топлива слито`.

Коли значення більше нуля, зберегти рядок події з `unit_chronology`.

## 13. Як знайти driver ID

Перевірити три джерела:

1. Звіт із таблицею `Поездки`, де є колонка водія.
2. `core/search_items` або resource-запити під час відкриття розділу водіїв.
3. Окремий report template для водіїв.

У Network шукати:

```text
driver
```

або:

```text
resource/
```

Для кожного водія потрібні:

- стабільний numeric ID;
- ім'я;
- код;
- інтервал прив'язки до unit.

Не будувати рейтинг лише за поточною прив'язкою водія. Потрібно знати,
хто був прив'язаний до автомобіля саме під час конкретної поїздки.

## 14. Як знайти Telegram chat ID

1. Створити нового Telegram-бота через BotFather.
2. Додати його в потрібний чат.
3. Надіслати в чат тестове повідомлення.
4. Викликати Telegram `getUpdates`.
5. Знайти:

```json
{
  "message": {
    "chat": {
      "id": 123456789,
      "title": "Fleet Reports"
    }
  }
}
```

6. Записати лише `chat.id` у реєстр.
7. Bot token залишити тільки у Vercel Environment Variables.

Для групи `chat.id` часто є від'ємним. Для forum topic додатково потрібен
`message_thread_id`.

## 15. Що перевірити зараз у трьох звітах

Цю перевірку потрібно один раз зробити вручну в інтерфейсі Moniterra.

### `.Отчет по топливу`

1. Вибрати групу `Брокінвест Групп, ТОВ`, якщо вона доступна у полі
   `Object`.
2. Вибрати `Yesterday` і натиснути `Execute`.
3. Якщо звіт виконався, зберегти без `sid`:
   - `report/exec_report` -> `Payload -> params`;
   - `report/apply_report_result` -> `Response`;
   - `report/select_result_rows` -> `Payload` і `Response`.
4. Якщо групу вибрати неможливо або звіт повертає помилку, достатньо
   зберегти screenshot і текст помилки. Це означає, що шаблон працює
   лише для окремого автомобіля.

### `.Поездки`

1. Вибрати один автомобіль, наприклад `KA2790BA / AA2544XC`.
2. Вибрати `Yesterday` і виконати звіт.
3. У Network зберегти три запити:
   - `exec_report`;
   - `apply_report_result`;
   - `select_result_rows` для кожної таблиці результату.
4. Особливо перевірити, чи є колонки:
   - водій;
   - початкова та кінцева адреса;
   - пробіг;
   - тривалість;
   - середня та максимальна швидкість.

### `Геозоны`

1. Вибрати той самий автомобіль і день.
2. Виконати звіт.
3. Зберегти `exec_report`, `apply_report_result` і
   `select_result_rows`.
4. Особливо перевірити, чи є:
   - назва геозони;
   - час входу та виходу;
   - тривалість перебування;
   - адреса або координати.

Для аналізу достатньо передати JSON цих запитів і відповідей. Токен,
`sid`, cookies та authorization headers передавати не потрібно.
